# v4.6.1 Quest 3 WebXR startup hotfix v2

This hotfix supersedes the first v4.6.1 startup recovery after a real Quest 3 report that the headset remained on the system loading screen.

## Root causes removed

1. Legacy v4.5 awaited one or two `isSessionSupported()` promises before `requestSession()`, risking loss of transient user activation.
2. The generic XR button preferred `immersive-ar` and requested depth sensing, hit testing, anchors and hand tracking together.
3. The renderer did not submit an XR frame until reference-space setup, shader compilation, buffer allocation, scene packing and audio setup were complete.
4. A partially-created session was not always ended after WebGL, layer, reference-space or renderer failures; the legacy fallback could then request a second immersive session.
5. The frame loop had no outer exception boundary or no-frame watchdog.
6. Repeated starts retained WebGL resources and canvases.
7. Late legacy installers could retake the visible XR button.

## New startup contract

- The VR button calls `requestSession('immersive-vr', { optionalFeatures: ['local-floor'] })` as the first asynchronous XR operation inside the trusted click.
- A separate MR button requests a minimal `immersive-ar` session. Depth sensing is opt-in only through the diagnostic API.
- A continuous clear-frame bootstrap begins immediately after the `XRWebGLLayer` is installed and continues while reference space and scene resources are prepared.
- Every post-session failure ends the acquired session and releases WebGL resources.
- First-frame and running-frame watchdogs prevent a silent, permanently active session.
- A capture-phase click owner prevents old handlers from stealing the transient activation.
- `__UPRS461__.audit()` exposes the exact startup stage, errors, support probe, granted features and event history.

## Verification

- 72 inline scripts: syntax clean.
- Application regression: 307/307 self-tests, 76/76 models, no runtime errors or console warnings.
- Mocked Quest-style successful VR and MR sessions reach `running` with the first XR call being `requestSession` and active user activation.
- `makeXRCompatible` rejection and first-frame timeout both end the created session.
- Request rejection does not invoke the legacy fallback.

Real headset validation remains required because automated Chromium cannot emulate the Meta compositor, Quest tracking or the device permission UI.
